# AIP PDF Reader PoC Milestones

## 목적

Foxit 기반 AIP PDF 열람 문제가 반복될 때, 기존 Microsoft Purview/Azure Information Protection 권한 체계를 우회하지 않는 읽기 전용 관문 뷰어 PoC를 검증한다.

이 문서는 두 단계를 분리한다.

- 1차: 회사 협의 전 기술 검증. 실제 회사 AIP 문서에 접근하지 않는다.
- 2차: 회사 승인 후 PoC 완성. 승인된 테스트 사용자와 테스트 AIP PDF만 사용한다.

핵심 원칙은 다음이다.

- Microsoft 회사 계정 로그인과 기존 Purview/AIP 권한을 그대로 따른다.
- 임의 복호화, 권한 우회, 보호 해제 도구를 만들지 않는다.
- PoC 기본 기능은 보기 전용이다.
- 복사, 다운로드, 인쇄, 텍스트 추출, 일반 PDF export는 기본 비활성화한다.
- 복호화된 PDF를 평문 파일로 남기지 않는 구조를 우선한다.

## 전체 아키텍처 가정

```text
사용자 PC 앱
 -> Microsoft Entra ID 로그인
 -> MIP SDK sidecar 또는 native wrapper
 -> Microsoft Purview/AIP 권한 확인
 -> 허용 시 복호화 스트림 획득
 -> PDF.js 또는 MuPDF 기반 읽기 전용 표시
 -> 감사 로그 기록
```

1차에서는 MIP SDK와 실제 AIP PDF 처리를 완성하지 않아도 된다. 대신 일반 PDF 뷰어, Microsoft 로그인, 앱 패키징, 보안 UX의 뼈대를 검증한다.

2차에서는 회사가 승인한 앱 등록, 테스트 사용자, 테스트 문서를 사용해 MIP SDK 기반 AIP PDF 열람을 검증한다.

## 범위 밖

다음은 이번 PoC의 기본 범위에서 제외한다.

- Foxit 전체 기능 대체
- PDF 편집
- AIP 보호 해제
- 일반 PDF로 내보내기
- 대량 문서 변환
- 서버 기반 문서 복호화
- 전문 DRM 수준의 화면 캡처 차단
- 전사 배포
- 모바일 앱
- macOS 지원. 초기 PoC는 Windows 우선으로 둔다.

## 1차: 회사 협의 전 기술 검증

### 목표

회사 AIP 문서와 Purview 권한에 접근하지 않고, PoC가 기술적으로 성립하는지 먼저 확인한다.

1차 완료 시점에는 보안팀/IT에 다음을 보여줄 수 있어야 한다.

- 일반 PDF를 자체 뷰어에서 안정적으로 표시한다.
- Microsoft 회사 계정 로그인 버튼과 인증 흐름이 동작한다.
- 앱이 읽기 전용 UX를 제공한다.
- 복사, 다운로드, 인쇄, 텍스트 추출 기능이 기본적으로 노출되지 않는다.
- 실제 AIP PDF 처리는 회사 승인 후 2차에서만 진행한다.

### 1.1 개발 골격 선택

권장 기본값:

```text
Electron + PDF.js + MSAL
```

선택 이유:

- PDF.js 통합이 빠르다.
- Microsoft 로그인 UI 구현이 쉽다.
- PoC 패키징과 배포가 비교적 단순하다.
- 나중에 MIP SDK sidecar를 붙이기 쉽다.

대안:

```text
Tauri + PDF.js webview + Rust bridge
```

Tauri는 앱 크기가 작고 native bridge가 깔끔하지만, 초기 속도는 Electron이 유리하다.

실물 테스트 성공조건:

- 개발자가 로컬에서 앱을 실행할 수 있다.
- 빈 화면이 아니라 PDF 열기 화면이 표시된다.
- 앱 버전, 빌드 시간, 실행 환경이 개발자 콘솔 또는 About 화면에 표시된다.
- Windows 테스트 PC에서 설치 또는 압축 실행이 가능하다.

실패조건:

- 앱 실행에 개발 환경 전체가 필요하다.
- 사용자 PC에서 실행 파일만으로 실행되지 않는다.
- PDF 뷰어가 브라우저 새 탭에 의존한다.

### 1.2 일반 PDF 뷰어 구현

기능:

- 로컬 일반 PDF 파일 열기
- 페이지 렌더링
- 페이지 이동
- 확대/축소
- 전체 페이지 수 표시
- 에러 PDF 표시

실물 테스트 파일:

- 1페이지 텍스트 PDF
- 20페이지 이상 일반 문서
- 표가 있는 PDF
- 이미지가 있는 PDF
- 암호가 걸린 일반 PDF
- 깨진 PDF
- 50MB 이상 대용량 PDF

성공조건:

- 정상 PDF는 3초 이내 첫 페이지가 표시된다.
- 20페이지 문서에서 페이지 이동이 가능하다.
- 확대/축소 후 텍스트와 이미지가 깨지지 않는다.
- 암호 PDF는 “지원하지 않음 또는 암호 필요” 상태를 명확히 표시한다.
- 깨진 PDF는 앱이 죽지 않고 오류 상태를 표시한다.
- 대용량 PDF를 열어도 UI가 5초 이상 완전히 멈추지 않는다.

실패조건:

- PDF 열기 실패 시 앱 프로세스가 종료된다.
- 페이지 이동 중 흰 화면으로 고정된다.
- 렌더링 실패 원인을 사용자와 개발자가 구분할 수 없다.

### 1.3 읽기 전용 UX 골격

기능:

- 다운로드 버튼 없음
- 저장 버튼 없음
- 인쇄 버튼 없음
- 텍스트 선택 기본 비활성화
- 컨텍스트 메뉴 제한
- 드래그 앤 드롭 저장 동작 없음

성공조건:

- 사용자가 UI에서 “저장”, “다른 이름 저장”, “다운로드”, “인쇄” 명령을 찾을 수 없다.
- `Ctrl+P`, `Ctrl+S` 입력 시 앱 내부에서 차단되거나 아무 동작도 하지 않는다.
- 마우스 우클릭 메뉴에 복사/저장 관련 항목이 노출되지 않는다.
- PDF.js 기본 toolbar가 그대로 노출되지 않는다.

실패조건:

- PDF.js 기본 다운로드/인쇄 버튼이 보인다.
- 단축키로 인쇄 또는 저장 대화상자가 열린다.
- 렌더링된 PDF가 임시 폴더에 일반 PDF 파일로 남는다.

주의:

이 단계의 차단은 PoC UX 차단이다. 강한 DRM 보장은 아니다. 보안팀 설명에서는 “권한 우회 방지 구조의 1차 UX”라고 표현한다.

### 1.4 Microsoft 로그인 기술 검증

기능:

- Microsoft 로그인 버튼
- 회사 계정 로그인
- 로그인 성공 후 사용자 표시
- 로그아웃
- 토큰 만료 시 재로그인

회사 협의 전에는 다음 두 가지 중 가능한 쪽으로 진행한다.

- 개발자 개인 테스트 테넌트 사용
- 회사 계정 로그인 화면까지만 확인하고, 관리자 승인이 필요한 경우 2차로 넘김

성공조건:

- 로그인 버튼을 누르면 Microsoft 로그인 창이 열린다.
- 로그인 성공 시 이름, 이메일, tenant id 또는 account id가 표시된다.
- 로그아웃 후 사용자 정보가 사라진다.
- 로그인 실패, 취소, 관리자 승인 필요 상태를 서로 다른 메시지로 표시한다.

실패조건:

- 로그인 실패 원인을 모두 “알 수 없는 오류”로만 표시한다.
- 토큰이 평문 로그에 출력된다.
- 로그인 성공 후에도 어떤 계정인지 확인할 수 없다.

보안조건:

- access token, refresh token, id token을 앱 로그에 남기지 않는다.
- 토큰 캐시 위치와 삭제 방법을 문서화한다.
- 스크린샷 공유 시 계정 식별정보를 가릴 수 있게 한다.

### 1.5 AIP 감지 자리 만들기

1차에서는 실제 AIP 복호화를 하지 않는다. 대신 파일 처리 파이프라인에 “보호 PDF 감지 지점”을 만든다.

기능:

- 파일 열기 요청
- 일반 PDF로 분기
- 보호 PDF 의심 시 “회사 승인 후 지원 예정” 메시지 표시
- 진단 로그에 파일 크기, 확장자, PDF header 여부만 기록

성공조건:

- 일반 PDF는 기존 뷰어로 열린다.
- 보호 PDF 또는 지원 불명 파일은 평문 변환 없이 차단 화면으로 간다.
- 파일 원문 일부를 로그에 남기지 않는다.

실패조건:

- 보호 PDF 의심 파일을 임의 파서로 열려고 시도한다.
- 실패한 파일 내용을 base64나 텍스트로 로그에 남긴다.

### 1.6 로컬 감사 로그 골격

1차 로그는 민감정보를 줄인 개발용 로그다.

권장 JSONL 예시:

```json
{"ts":"2026-04-16T00:00:00Z","event":"open_pdf","user":"user@example.com","file_name":"sample.pdf","file_size":12345,"result":"success","reason":null,"app_version":"0.1.0"}
```

성공조건:

- PDF 열기 성공/실패가 JSONL로 기록된다.
- 로그인 성공/실패가 JSONL로 기록된다.
- 토큰, 파일 내용, PDF 텍스트는 로그에 기록되지 않는다.
- 로그 파일 위치가 문서화되어 있다.

실패조건:

- 로그에 access token, refresh token, PDF 텍스트, 문서 본문이 남는다.
- 실패 원인이 기록되지 않아 디버깅이 불가능하다.

### 1.7 1차 산출물

필수 산출물:

- 실행 가능한 앱 또는 개발 빌드
- 일반 PDF 테스트 결과표
- Microsoft 로그인 테스트 결과표
- 읽기 전용 UX 차단 테스트 결과표
- 2차 협의용 보안 설명 1페이지
- 회사 요청사항 목록

1차 완료 판정:

```text
일반 PDF 6종 테스트 통과
Microsoft 로그인 성공 또는 관리자 승인 필요 상태까지 재현
저장/인쇄/다운로드 UI 미노출
토큰/문서 원문 로그 미기록 확인
2차 회사 협의 문서 준비 완료
```

## 2차: 회사 승인 후 읽기 전용 AIP PDF PoC

### 목표

회사 승인된 앱 등록, 테스트 사용자, 테스트 AIP PDF를 사용해 공식 MIP SDK 기반 읽기 전용 열람이 가능한지 검증한다.

2차 완료 시점에는 다음 질문에 답할 수 있어야 한다.

- Foxit 없이도 승인된 사용자가 AIP PDF를 열 수 있는가?
- 권한 없는 사용자는 차단되는가?
- 평문 PDF가 디스크에 남지 않는가?
- 저장/복사/인쇄/추출이 기본 차단되는가?
- 감사 로그로 누가 무엇을 시도했는지 확인 가능한가?
- 회사 운영 제품으로 확장할 가치가 있는가?

### 2.1 회사 승인 준비

회사 측 필요 작업:

- Entra ID 앱 등록 또는 기존 등록 앱 사용 승인
- redirect URI 확정
- 테스트 사용자 3~5명 지정
- 테스트 AIP PDF 3~5개 제공
- 관리자 동의 필요 권한 확인
- PoC 기간 승인
- 감사 로그 항목 승인
- 배포 방식 승인

개발 측 제공 자료:

- 앱 이름과 목적
- 요청 권한 목록
- redirect URI
- 테스트 사용자 목록 초안
- 저장/복사/인쇄/추출 금지 범위
- 평문 파일 미저장 설계
- 로그 샘플

성공조건:

- 앱 client id가 확정된다.
- 테스트 사용자가 로그인 가능하다.
- 테스트 AIP PDF가 준비된다.
- 보안팀/IT가 “제한 PoC” 범위를 승인한다.

실패조건:

- 운영 문서를 임의로 테스트에 사용한다.
- 관리자 동의 범위가 불명확하다.
- 앱 등록 주체와 운영 책임자가 정해지지 않는다.

### 2.2 MIP SDK sidecar 연동

구조:

```text
Electron UI
 -> local sidecar process
 -> MIP SDK
 -> auth token callback
 -> protected file handler
 -> decrypted stream or temp-protected rendering input
```

기능:

- sidecar 실행/종료 관리
- 앱에서 sidecar health check
- 로그인 토큰 전달 또는 토큰 획득 콜백 연결
- MIP SDK 초기화
- 파일 보호 상태 확인
- 권한 확인
- 허용 시 렌더링용 입력 제공

성공조건:

- 앱 시작 시 sidecar 상태가 확인된다.
- sidecar 죽음, SDK 초기화 실패, 인증 실패가 구분되어 표시된다.
- 일반 PDF 경로와 AIP PDF 경로가 분리된다.
- sidecar 로그에 토큰과 문서 본문이 남지 않는다.

실패조건:

- sidecar 실패가 앱 전체 크래시로 이어진다.
- sidecar가 복호화된 PDF를 고정 경로에 저장한다.
- SDK 초기화 실패 원인을 알 수 없다.

### 2.3 AIP PDF 권한 확인

테스트 문서 세트:

- A: 테스트 사용자 열람 허용
- B: 테스트 사용자 열람 불가
- C: 만료된 문서
- D: 열람 가능하지만 인쇄 금지
- E: 열람 가능하지만 복사 금지

기능:

- 보호 PDF 판별
- 사용자 권한 확인
- 권한 있음/없음/만료/정책 오류/네트워크 오류 구분

성공조건:

- A 문서는 열린다.
- B 문서는 열리지 않고 권한 없음 메시지를 표시한다.
- C 문서는 열리지 않고 만료 메시지를 표시한다.
- 네트워크 차단 시 정책 확인 실패 메시지를 표시한다.
- 로그에 각 결과가 구분되어 남는다.

실패조건:

- 권한 없는 문서가 열린다.
- 만료 문서가 일반 오류로만 표시된다.
- 네트워크 오류와 권한 없음이 구분되지 않는다.

### 2.4 읽기 전용 렌더링

기능:

- 허용된 AIP PDF 첫 페이지 표시
- 페이지 이동
- 확대/축소
- 대용량 문서 처리
- 렌더링 실패 복구

성공조건:

- 권한 있는 AIP PDF의 첫 페이지가 5초 이내 표시된다. 대용량은 별도 기준을 둔다.
- 페이지 이동이 가능하다.
- 20페이지 이상 문서에서 페이지 번호가 맞는다.
- 확대/축소 후 렌더 품질이 실사용 가능하다.
- 문서를 닫으면 렌더링 상태와 메모리 참조가 해제된다.

실패조건:

- 첫 페이지 표시 후 다음 페이지 이동이 안 된다.
- 닫은 문서를 다시 열 때 이전 문서 내용이 남는다.
- 앱 재시작 후 평문 PDF가 임시 폴더에 남아 있다.

### 2.5 저장/복사/인쇄/추출 차단

기능:

- 다운로드 버튼 없음
- 저장 버튼 없음
- 인쇄 버튼 없음
- 텍스트 선택 차단 또는 권한 기반 비활성화
- `Ctrl+S`, `Ctrl+P`, `Ctrl+C` 차단
- 우클릭 메뉴 차단
- drag-out 차단

성공조건:

- UI에서 다운로드/저장/인쇄 기능을 찾을 수 없다.
- 단축키로 OS 저장/인쇄 대화상자가 뜨지 않는다.
- PDF 텍스트를 선택해 복사할 수 없다.
- 렌더링 캔버스를 drag해서 파일로 끌어낼 수 없다.
- 개발자 로그나 앱 로그에 PDF 텍스트가 남지 않는다.

실패조건:

- PDF.js 기본 toolbar가 노출된다.
- `Ctrl+P`로 인쇄 대화상자가 열린다.
- `Ctrl+S`로 복호화된 문서 저장이 가능하다.
- 복호화된 PDF나 텍스트가 임시 파일에 남는다.

주의:

PoC는 일반 데스크톱 화면 캡처를 완전히 막는 DRM 제품이 아니다. 이 한계는 보안팀에 명시한다.

### 2.6 평문 파일 미저장 검증

검증 대상:

- 앱 작업 디렉터리
- OS temp 디렉터리
- 사용자 Downloads/Documents
- 앱 캐시 디렉터리
- sidecar 작업 디렉터리
- 로그 디렉터리

성공조건:

- AIP PDF 열람 후 일반 PDF 확장자의 복호화 산출물이 남지 않는다.
- 로그에 PDF 본문 텍스트가 남지 않는다.
- 앱 종료 후 임시 렌더링 산출물이 정리된다.
- 필요한 캐시가 있다면 암호화 여부, TTL, 삭제 방법이 문서화된다.

실패조건:

- 복호화된 PDF 파일이 temp에 남는다.
- 파일명만 바꾼 평문 PDF가 캐시에 남는다.
- 로그에 문서 본문 일부가 남는다.

실물 테스트 방법:

```text
1. 테스트 AIP PDF 열기
2. 문서 닫기
3. 앱 종료
4. temp/cache/log/downloads 경로 스캔
5. 파일명, PDF header, 테스트 문서 고유 문자열 검색
```

필수 이벤트:

- open_aip_pdf_success
- open_aip_pdf_denied

권장 필드:

  "user": "user@example.com",
  "tenant_id": "tenant-id",
  "file_name_hash": "sha256...",

  "policy_result": "view_allowed",
  "blocked_capabilities": ["print", "download", "copy", "extract"],

성공조건:

- 열람 성공/실패가 구분되어 기록된다.
- 차단된 저장/인쇄/복사 시도도 기록된다.
- 토큰, PDF 본문, 복호화 산출물 경로가 기록되지 않는다.
- 회사가 요구하면 SIEM 또는 서버 전송으로 확장 가능한 포맷이다.

실패조건:

- 누가 어떤 시도를 했는지 추적할 수 없다.
- 로그가 민감정보를 과하게 포함한다.
- 로그가 사용자가 쉽게 삭제 가능한 유일한 위치에만 있다.

### 2.8 패키징과 배포 테스트

초기 기본값:

- Windows x64
- 내부 테스트용 설치 파일 또는 압축 배포
- 코드서명은 회사 요구에 맞춤
- 자동 업데이트는 제외

성공조건:

- 테스트 PC 2대 이상에서 설치 또는 실행된다.
- 개발 도구 없이 실행된다.
- 앱 삭제 후 캐시/로그 처리 방식이 문서화되어 있다.
- 버전 정보가 앱과 로그에 남는다.

실패조건:

- 개발자 PC에서만 동작한다.
- 설치 후 sidecar 경로를 찾지 못한다.
- 테스트 사용자마다 수동 설정이 과도하게 필요하다.

### 2.9 사용자 실물 시나리오 테스트

시나리오 1: 권한 있는 사용자 열람

- Given: 테스트 사용자 U1은 문서 A 열람 권한이 있다.
- When: U1이 앱에서 문서 A를 연다.
- Then: 첫 페이지가 표시되고 페이지 이동이 가능하다.
- And: 저장/인쇄/복사 기능은 보이지 않는다.
- And: 감사 로그에 success가 남는다.

시나리오 2: 권한 없는 사용자 차단

- Given: 테스트 사용자 U2는 문서 B 열람 권한이 없다.
- When: U2가 문서 B를 연다.
- Then: 문서는 표시되지 않는다.
- And: 권한 없음 메시지가 표시된다.
- And: 감사 로그에 denied가 남는다.

시나리오 3: 만료 문서 차단

- Given: 문서 C는 만료되었다.
- When: 테스트 사용자가 문서 C를 연다.
- Then: 문서는 표시되지 않는다.
- And: 만료 메시지가 표시된다.
- And: 감사 로그에 expired가 남는다.

시나리오 4: 인쇄 시도 차단

- Given: 문서 D는 열람 가능하지만 인쇄 금지다.
- When: 사용자가 `Ctrl+P`를 누른다.
- Then: OS 인쇄 대화상자가 열리지 않는다.
- And: 감사 로그에 blocked_action print가 남는다.

시나리오 5: 복사 시도 차단

- Given: 문서 E는 열람 가능하지만 복사 금지다.
- When: 사용자가 텍스트 선택 또는 `Ctrl+C`를 시도한다.
- Then: 클립보드에 문서 텍스트가 들어가지 않는다.
- And: 감사 로그에 blocked_action copy가 남는다.

시나리오 6: 네트워크 장애

- Given: 앱은 로그인되어 있다.
- When: 네트워크를 끊고 AIP PDF를 연다.
- Then: 정책 확인 실패 또는 오프라인 미지원 메시지가 표시된다.
- And: 문서는 임의로 열리지 않는다.

시나리오 7: 앱 종료 후 잔존 파일 검사

- Given: 권한 있는 AIP PDF를 열람했다.
- When: 문서를 닫고 앱을 종료한다.
- Then: temp/cache/downloads/logs에 평문 PDF 또는 문서 본문이 남지 않는다.

### 2.10 2차 완료 판정

2차 PoC 완료 기준:

```text
승인된 테스트 사용자 로그인 성공
권한 있는 AIP PDF 열람 성공
권한 없는 AIP PDF 차단 성공
만료/정책/네트워크 오류 구분 성공
저장/복사/인쇄/추출 기본 차단 성공
평문 PDF 잔존 파일 없음 확인
감사 로그 이벤트 기록 확인
테스트 PC 2대 이상 실행 확인
회사 보안팀에 한계와 다음 단계 설명 가능
```

2차 PoC 실패 판정:

```text
권한 없는 문서가 열린다
복호화된 PDF가 디스크에 남는다
토큰 또는 문서 본문이 로그에 남는다
저장/인쇄/복사 차단이 쉽게 우회된다
테스트 PC에서 재현성이 없다
MIP SDK 연동이 회사 승인 정책과 충돌한다
```

## 리스크와 대응

### 앱 승인 지연

리스크:

- 회사 테넌트에서 사용자 동의가 막혀 있을 수 있다.
- 관리자 동의가 필요할 수 있다.

대응:

- 1차에서는 로그인 화면과 관리자 승인 필요 상태까지만 확인한다.
- 2차 전에 앱 등록 요청서를 준비한다.

### MIP SDK 통합 난도

리스크:

- C++ SDK, 인증 콜백, native dependency가 어렵다.

대응:

- UI 앱과 MIP 처리를 sidecar로 분리한다.
- SDK 샘플을 먼저 단독 실행한다.
- 실패 원인을 UI/sidecar/SDK/auth/policy로 구분해 로깅한다.

### 평문 파일 잔존

리스크:

- 렌더링 편의를 위해 임시 PDF를 만들면 보안 리스크가 생긴다.

대응:

- 메모리 스트림 우선.
- 임시 파일이 불가피하면 암호화, TTL, 종료 시 삭제, 잔존 검사 자동화를 요구한다.

### DRM 오해

리스크:

- 보안팀이 화면 캡처 차단까지 기대할 수 있다.

대응:

- PoC 범위를 “기존 권한 체계 준수 읽기 전용 관문”으로 정의한다.
- 화면 캡처 완전 차단은 범위 밖이라고 명시한다.

### Foxit 완전 대체 요구

리스크:

- 편집, 주석, 서명, 양식, 인쇄까지 요구되면 범위가 폭증한다.

대응:

- PoC는 문제 재현 문서의 안정적 열람에 한정한다.
- 성공 후 확장 기능은 별도 우선순위로 평가한다.

## Go/No-Go 기준

### 1차 종료 후 Go

다음 조건을 만족하면 회사 협의로 넘어간다.

- 일반 PDF 뷰어가 실물 파일로 안정 동작한다.
- Microsoft 로그인 또는 승인 필요 상태가 재현된다.
- 읽기 전용 UX가 작동한다.
- 보안 설명 자료가 준비된다.
- 개발팀이 MIP SDK sidecar 통합 경로를 확인했다.

### 1차 종료 후 No-Go

다음이면 2차로 가지 않는다.

- PDF 뷰어 안정성이 낮다.
- 로그인 흐름을 설명할 수 없다.
- 저장/인쇄/복사 차단이 PoC 수준에서도 허술하다.
- 회사 협의 문장으로 방어 가능한 범위 정의가 없다.

### 2차 종료 후 Go

다음이면 운영 검토로 넘어간다.

- 권한 있는 사용자 열람 성공률이 높다.
- 권한 없는 사용자 차단이 확실하다.
- 평문 파일 잔존이 없다.
- 감사 로그가 충분하다.
- Foxit 문제 문서에서 체감 개선이 있다.
- 보안팀이 남은 한계를 수용할 수 있다.

### 2차 종료 후 No-Go

다음이면 운영화를 중단하거나 범위를 재설계한다.

- MIP SDK 연동이 안정적이지 않다.
- 보안상 평문 파일 잔존을 피할 수 없다.
- 권한/정책 오류가 자주 오판된다.
- 회사가 요구하는 DRM 수준과 PoC 구조가 맞지 않는다.
