# AIP PDF 읽기 전용 관문 1차 순차개발 보고

## 시작 개요

이번 개발은 Foxit 기반 AIP PDF 열람 문제를 줄일 수 있는 자작 관문 뷰어의 가능성을 확인하기 위한 1차 기술검증입니다. 회사 AIP 문서나 실제 Purview 권한에는 아직 접근하지 않고, 일반 PDF 뷰어와 읽기 전용 사용 흐름, Microsoft 로그인 골격, 감사 로그, 2차 협의 자료까지 준비하는 것을 목표로 합니다.

전체 흐름은 먼저 데스크톱 앱 골격을 만들고, 일반 PDF가 안정적으로 열리는지 확인한 뒤, 저장·인쇄·복사 같은 기능이 노출되지 않는 읽기 전용 관문 UX를 붙이는 방식으로 진행합니다. 이후 Microsoft 로그인 골격과 AIP 감지 자리를 마련하고, 회사 협의에 바로 사용할 수 있는 요청 자료와 테스트 결과 기준을 정리합니다.

## 실행 체크리스트

- [x] 앱 골격을 만들고 테스트 PC에서 실행 가능한 형태로 준비합니다.
- [x] 일반 PDF 뷰어 기능을 붙이고 정상·대용량·오류 PDF 흐름을 검증합니다.
- [x] 저장, 인쇄, 다운로드, 복사 기능이 기본 노출되지 않도록 읽기 전용 UX를 구현합니다.
- [x] Microsoft 로그인 골격을 붙여 실제 앱 승인 전에도 상태를 구분할 수 있게 합니다.
- [ ] AIP 감지 자리와 로컬 감사 로그를 구현합니다.
- [ ] 회사 협의용 1페이지 설명과 1차 최종 테스트 결과를 정리합니다.

## 대화 배경 요약

사용자께서는 Foxit의 AIP PDF 열람 경험이 좋지 않아, 회사 정책을 우회하지 않는 대체 PoC가 가능한지 확인하고자 하셨습니다. 논의 결과, 공식 Microsoft MIP SDK를 전제로 한 읽기 전용 관문 앱이 현실적인 방향이며, 먼저 회사 협의 전 기술검증을 진행하고 이후 회사 승인 기반 PoC로 넘어가는 단계형 접근이 적절하다고 정리했습니다.

## 1차-1 앱 골격 완료

첫 마일스톤에서는 데스크톱 앱이 실제로 시작될 수 있는 최소 골격을 마련했습니다. 사용자가 앱을 열면 1차 기술검증 목적, 비활성 상태의 PDF 열기와 Microsoft 로그인 진입점, 앱 상태 정보를 확인할 수 있습니다.

검증은 두 갈래로 진행했습니다. 코드 수준에서는 앱 정보 생성 로직과 렌더러 화면 골격 테스트를 실행했고 모두 통과했습니다. 행동 수준에서는 Electron 실행 파일 버전을 확인해 실제 데스크톱 앱 런타임이 설치되었음을 확인했습니다.

아직 PDF 열기와 로그인은 의도적으로 비활성 상태입니다. 다음 마일스톤에서 일반 PDF 뷰어 기능을 붙이면서 이 진입점을 실제 동작으로 전환합니다.

## 1차-2 일반 PDF 뷰어 완료

두 번째 마일스톤에서는 일반 PDF 파일을 선택하고 PDF.js로 렌더링하는 흐름을 붙였습니다. 앱은 운영체제 파일 선택 창에서 PDF만 선택하게 하고, 선택된 파일은 파일 경로가 아니라 메모리 바이트로 렌더러에 전달합니다.

사용자 입장에서는 PDF 열기 버튼, 페이지 이동, 확대와 축소, 현재 페이지 표시가 생겼습니다. 깨진 PDF나 열 수 없는 PDF는 문서를 표시하지 않고 오류 상태로 전환하도록 했습니다.

검증은 `pnpm test`로 진행했습니다. PDF.js 연결, 파일 선택 제한, 메모리 바이트 렌더링 경로, 화면 컨트롤 존재 여부를 확인했고, 테스트 안에서 만든 실제 1페이지 PDF 바이트를 PDF.js가 읽어 페이지 수와 크기를 반환하는 것도 확인했습니다.

## 1차-3 읽기 전용 UX 차단 완료

세 번째 마일스톤에서는 저장, 인쇄, 복사, 우클릭 메뉴, 드래그 동작을 기본 차단하는 읽기 전용 UX를 구현했습니다. 사용자가 PDF를 보는 화면에는 다운로드나 인쇄 버튼이 없고, 단축키나 우클릭으로도 해당 동작이 진행되지 않도록 렌더러 차단 로직을 설치했습니다.

이 구현은 강한 DRM을 의미하지 않습니다. PoC의 목표는 회사 협의 전에 “읽기 전용 관문으로 설계되어 있다”는 사용자 경험과 기본 방어선을 보여주는 것입니다. 화면 캡처 완전 차단은 이번 1차 범위 밖으로 유지합니다.

검증은 `pnpm test`로 진행했고, 저장·다운로드·인쇄 명령이 화면에 노출되지 않는지, `Ctrl+P`, `Ctrl+S`, `Ctrl+C`, 우클릭, 드래그, 브라우저 인쇄 이벤트 차단 코드가 존재하는지 확인했습니다.

## 1차-4 Microsoft 로그인 골격 완료

네 번째 마일스톤에서는 Microsoft 회사 계정 로그인으로 넘어가기 위한 앱 내부 골격을 만들었습니다. 아직 회사 Entra 앱 승인이 없으므로 기본 상태는 “앱 승인 정보 필요”로 표시됩니다. 환경변수로 승인된 client id와 tenant 정보를 제공하면 Microsoft identity platform 로그인 URL을 열 수 있습니다.

이번 단계에서는 토큰을 받거나 저장하지 않습니다. 1차 목적은 회사 협의 전에 로그인 진입점, 승인 정보 필요 상태, 승인 후 로그인 창을 열 수 있는 구조를 보여주는 것입니다.

검증은 `pnpm test`로 진행했습니다. 승인 정보가 없을 때의 상태, 승인 정보가 있을 때의 설정 반영, Microsoft authorize URL 생성, 토큰 API가 preload에 노출되지 않는 점을 확인했습니다.
